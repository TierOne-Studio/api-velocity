# ADR-008: No AI-attribution trailers in commits, PRs, issues, or releases

**Status:** Accepted
**Date:** 2026-04-30
**Deciders:** core engineering team

## Context

Claude Code's default commit-message guidance injects `Co-Authored-By: Claude <noreply@anthropic.com>` and a `🤖 Generated with [Claude Code]` footer into commits, PR descriptions, and similar artifacts. Many AI tooling defaults do the same (different vendor names, same shape).

The repo's policy is that the human git user owns the artifact. Tool used to draft it is irrelevant to the published commit/PR.

## Decision

Commit messages, PR titles, PR descriptions, issue comments, and release notes MUST NEVER contain `Co-Authored-By: Claude`, `🤖 Generated with [Claude Code]`, "Generated by Anthropic", or any similar AI-attribution trailer/footer. The git author/committer is the human user; assistant identity does not appear in the artifact.

This rule overrides any tool default that would inject such a trailer.

## Alternatives considered

- **Allow opt-in attribution.** Rejected: the user has stated a repo-wide policy; opt-in produces inconsistent artifacts that depend on which contributor's local config is active that day.
- **Allow attribution but redact at PR-merge time.** Rejected: relies on a process step that doesn't exist and adds friction without value.
- **Tolerate tool defaults silently.** Rejected: produces commit/PR artifacts that misrepresent authorship.

## Consequences

- **Positive:** clean git history; commits read as authored by the human listed in `Author:`. No noise in `git log`.
- **Negative:** must remember to override the tool default every time; if a tool injects the trailer, the commit must be amended (or a new commit created) before pushing.
- **Follow-ups:** wired in `CLAUDE.md` § P0.1 and `git-workflow` SKILL.md "Hard rules". Acceptance assertion T66 verifies the rule stays wired AND defensively asserts the current HEAD commit message does not contain the forbidden trailers.
